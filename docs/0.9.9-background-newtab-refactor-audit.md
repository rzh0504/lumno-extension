# Lumno 0.9.9 Background / Newtab Refactor Audit

## Goal

This audit is the first gate before continuing to split `src/background/background.js`
and `src/newtab/newtab.js`.

The goal is not to make files smaller at any cost. The goal is to improve
maintainability while preserving:

- exact UI structure, styling, spacing, animation, and interaction timing;
- exact search, provider, tab, favicon, bookmark, recent-site, and PiP behavior;
- parity between normal-page overlay and newtab search.

## Runtime Message Route Audit

`background.js` currently has one long `chrome.runtime.onMessage` branch around
line 2063. The actions can be grouped without changing behavior:

| Group | Actions | First step |
| --- | --- | --- |
| Tab visibility/switching | `switchToTab`, `reportTabVisible`, `getTabsForOverlay`, `trackSearchTab`, `closeOtherTabsForOverlay` | Route grouping only; behavior touches ranking and UI selection. |
| Page hotkey | `getShowSearchShortcut`, `getCopyCurrentUrlCommandShortcut`, `triggerShowSearchFromPageHotkey` | Keep close to command handlers until shortcut fallbacks are stable. |
| PiP ownership/main-world | `pipRequestOwnership`, `pipReleaseOwnership`, `siteTryEnterPiPInMainWorld`, `iqiyiTryEnterPiPInMainWorld`, `iqiyiSetupAutoPiPInMainWorld`, `forceExitPiPInMainWorld`, `ytForceExitPiPInMainWorld` | Good candidate for a background PiP message module after router shape exists. |
| Search | `searchOrNavigate`, `getSearchSuggestions`, `recordSearchSuggestionSelection`, `deleteHistoryUrl` | Do not split until fixtures compare overlay/newtab results. |
| Site search / AI | `getSiteSearchProviders`, `runSiteSearchProviderQuery` | Worth extracting provider trigger/match helpers first because code is duplicated in overlay and newtab. |
| Shortcuts / locale / permissions | `getShortcutRules`, `getLocaleMessages`, `getFileSchemeAccessStatus` | Low-risk handlers, but keep async `sendResponse` semantics explicit. |
| Extension pages | `openOptionsPage`, `openExtensionShortcutsPage`, `openBookmarkManager`, `createTab`, `openNewTab`, `openExtensionDetailsPage` | Low-risk service helper candidate. |
| Favicon service | `resolveFaviconCandidates`, `getFaviconData` | Medium-risk: good boundary, but verify local-network/favicon proxy guards. |

Suggested router shape:

1. First convert the long branch into a local action table inside
   `background.js`, preserving every handler body and every `return true`.
2. Then move only self-contained groups to `src/background/*` modules.
3. Avoid a router module that needs dozens of implicit globals. Use explicit
   dependency injection if a handler is moved.

## Extraction Candidates

### Do Now

These have real maintenance value and relatively clear boundaries:

| Candidate | Proposed target | Why it is worth doing | Risk |
| --- | --- | --- | --- |
| Runtime message routing table | keep local first, optional `src/background/message-router.js` later | Removes the long `if/else` action chain from day-to-day reading. | Low if handler bodies stay unchanged. |
| Site-search trigger/match helpers | `src/shared/site-search-utils.js` or extend `search-utils.js` | `newtab.js` and overlay code both define provider host matching, trigger matching, inline provider query, display-name key maps, and top-site/provider matching. | Medium: must preserve `keyword + Tab -> Enter` exactly. |
| Provider display-name mapping | shared provider helper | Currently repeated across `background.js`, `newtab.js`, and `options.js`, and recent AI keys increased duplication. | Low. |
| Extension page/open helpers | `src/background/extension-pages.js` | Mostly Chrome API wrappers and URL builders. | Low. |
| File access / newtab fallback helpers | background helper module | Pure URL construction plus permission check wrapper. | Low-medium because local-file fallback copy must stay exact. |
| PiP main-world handlers | `src/background/pip-main-world.js` | Very large message bodies, mostly isolated from search/newtab. | Medium-high: needs targeted PiP smoke checks. |

### Do Later

These are useful, but should wait until route/shared helper work lands:

| Candidate | Why wait |
| --- | --- |
| Favicon resolver service | It touches local-network blocking, `chrome://favicon2`, fallback order, and data cache behavior. Worth doing, but tests should come first. |
| Newtab recent-site store | Hidden/pinned normalization and storage helpers can move, but rendering and hover actions should stay local at first. |
| Newtab bookmark data store | Data tree/cache functions can move after recent-site extraction pattern is proven. |
| Newtab page notice | Small and self-contained, but moving it gives less value than route/provider work. |
| Folder icon morph animation | It is visually sensitive and not shared. Moving it may reduce file size, but it does not improve the cross-entry maintenance problem much. |

### Do Not Split Yet

These should stay put until there is a stronger reason:

- overlay render and selection state inside `toggleBlackRectangle`;
- newtab `renderSuggestions`, `applySearchSuggestionHighlight`, and
  `updateSelection`;
- layout measurement such as `updateSearchEntryLayout`,
  `updateBookmarkSectionPosition`, bottom dock sizing, and body padding;
- AI beam/sweep geometry and input-mode prefix animation;
- site-search prefix pill rendering, icon/theming, padding calculation, and
  active site-search suggestion mode from `b05de9f`;
- IME/autocomplete focus-session behavior;
- any style block where extraction would change script load order, first paint,
  or inline style write order.

## Shared Overlay / Newtab Code Audit

Good shared candidates:

- provider host extraction and host equality;
- provider trigger/key/alias matching;
- provider display-name message-key mapping;
- `isInteractiveSiteSearchProvider` fallback list;
- inline provider query parsing;
- strong navigation/top-site promotion wrappers, if dependency callbacks remain
  explicit;
- suggestion scroll behavior already lives in `suggestion-navigation.js`;
- URL guards already live in `url-guards.js`;
- provider loading already lives in `site-search-store.js`.

Not good shared candidates yet:

- actual DOM rendering for suggestion rows;
- hover/selected visual-state writes;
- `siteSearchState` / `inlineSearchState` lifecycle;
- input padding and prefix geometry, especially the `b05de9f` prefix pill
  contract;
- autocomplete suppression after delete;
- bookmark/recent card UI.

Rule of thumb: if the helper can be tested with plain objects, share it. If it
depends on DOM, first paint, focus, or animation state, keep it local for now.

## Recommended Implementation Order

1. Settle the current AI/provider work:
   - `node --check` for touched JS;
   - JSON parse for `assets/data/site-search.json` and locale files;
   - confirm no missing runtime files in `manifest` / package whitelist.
2. Convert `background.js` message handling into a grouped local route table.
   This is a readability step, not a behavior split.
3. Extract provider display-name and provider trigger/match helpers to shared
   code, then update overlay and newtab callers together.
4. Add small plain-object tests for provider helpers.
5. Only then move one background group, preferably extension-page helpers or
   PiP main-world handlers.
6. Start `!important` reduction in newtab suggestion state only, using
   class/data-state and checking visual parity.
7. Leave overlay shell/geometry important untouched until the inner shadow-root
   state styles are classed and verified.

## Verification Gates

For every implementation slice:

- `node --check` for touched JS files;
- JSON parse for changed JSON;
- manifest/local resource existence check;
- search/provider fixtures when provider or search behavior moves;
- dual-entry manual check only when the slice touches UI behavior:
  normal-page overlay plus newtab.

For search/provider changes, the regression set must include:

- plain query: `lumno`;
- provider trigger: `gm`;
- provider trigger plus query: `gm hello`;
- AI provider trigger such as `gpt hello`;
- active site-search mode after `Tab`: one provider suggestion, correct prefix
  pill text/style, input padding, and caret color;
- direct URL/navigation-like input;
- history/bookmark/top-site result that can be switched or opened.

## Decision

There is enough reason to continue refactoring, but not by blindly slicing large
files. The highest-value next work is:

1. route table in `background.js`;
2. shared site-search/provider matching helpers;
3. provider display-name mapping;
4. targeted newtab important cleanup.

Moving visual renderers or layout code should wait until these lower-risk
boundaries are done.
