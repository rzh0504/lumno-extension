# Lumno 0.9.9 Background / Newtab Refactor Audit

## Goal

This audit is the first gate before continuing to split `src/background/background.js`
and `src/newtab/newtab.js`.

The goal is not to make files smaller at any cost. The goal is to improve
maintainability while preserving:

- exact UI structure, styling, spacing, animation, and interaction timing;
- exact search, provider, tab, favicon, bookmark, recent-site, and PiP behavior;
- parity between normal-page overlay and newtab search.

## Extraction Candidates

### Do Now

These have real maintenance value and relatively clear boundaries:

| Candidate | Proposed target | Why it is worth doing | Risk |
| --- | --- | --- | --- |
| PiP main-world handlers | `src/background/pip-main-world.js` | Very large message bodies, mostly isolated from search/newtab. | Done with wrapper stub tests; manual PiP smoke remains optional when validating on real video pages. |

### Do Later

These are useful, but should wait until route/shared helper work lands:

| Candidate | Why wait |
| --- | --- |
| Favicon resolver service | It touches local-network blocking, `chrome://favicon2`, fallback order, and data cache behavior. Worth doing, but tests should come first. |
| Newtab recent-site store | Hidden/pinned normalization and storage helpers can move, but rendering and hover actions should stay local at first. |
| Newtab bookmark data store | Data tree/cache functions can move after recent-site extraction pattern is proven. |
| Newtab page notice | Small and self-contained, but moving it gives less value than route/provider work. |
| Folder icon morph animation | It is visually sensitive and not shared. Moving it may reduce file size, but it does not improve the cross-entry maintenance problem much. |

### Do Not Split Yet

These should stay put until there is a stronger reason:

- overlay render and selection state inside `toggleBlackRectangle`;
- newtab `renderSuggestions`, `applySearchSuggestionHighlight`, and
  `updateSelection`;
- layout measurement such as `updateSearchEntryLayout`,
  `updateBookmarkSectionPosition`, bottom dock sizing, and body padding;
- AI beam/sweep geometry and input-mode prefix animation;
- site-search prefix pill rendering, icon/theming, padding calculation, and
  active site-search suggestion mode from `b05de9f`;
- IME/autocomplete focus-session behavior;
- any style block where extraction would change script load order, first paint,
  or inline style write order.

## Shared Overlay / Newtab Code Audit

Good shared candidates:

- provider host extraction and host equality;
- provider trigger/key/alias matching;
- provider display-name message-key mapping;
- `isInteractiveSiteSearchProvider` fallback list;
- inline provider query parsing;
- strong navigation/top-site promotion wrappers, if dependency callbacks remain
  explicit;
- suggestion scroll behavior already lives in `suggestion-navigation.js`;
- URL guards already live in `url-guards.js`;
- provider loading already lives in `site-search-store.js`.

Not good shared candidates yet:

- actual DOM rendering for suggestion rows;
- hover/selected visual-state writes;
- `siteSearchState` / `inlineSearchState` lifecycle;
- input padding and prefix geometry, especially the `b05de9f` prefix pill
  contract;
- autocomplete suppression after delete;
- bookmark/recent card UI.

Rule of thumb: if the helper can be tested with plain objects, share it. If it
depends on DOM, first paint, focus, or animation state, keep it local for now.

## Verification Gates

For every implementation slice:

- `node --check` for touched JS files;
- JSON parse for changed JSON;
- manifest/local resource existence check;
- search/provider fixtures when provider or search behavior moves;
- dual-entry manual check only when the slice touches UI behavior:
  normal-page overlay plus newtab.

For search/provider changes, the regression set must include:

- plain query: `lumno`;
- provider trigger: `gm`;
- provider trigger plus query: `gm hello`;
- AI provider trigger such as `gpt hello`;
- active site-search mode after `Tab`: one provider suggestion, correct prefix
  pill text/style, input padding, and caret color;
- direct URL/navigation-like input;
- history/bookmark/top-site result that can be switched or opened.

## Decision

There is enough reason to continue refactoring, but not by blindly slicing large
files. The highest-value next work is:

1. targeted newtab important cleanup;
2. fixture-backed search/provider split candidates;
3. later visual-state class conversion after targeted UI parity checks.

Moving visual renderers or layout code should wait until these lower-risk
boundaries are done.
